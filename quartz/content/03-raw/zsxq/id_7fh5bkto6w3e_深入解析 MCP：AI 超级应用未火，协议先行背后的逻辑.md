今天，我们来探讨一个近期备受关注的技术——**MCP（Model Context Protocol，模型上下文协议），**系统性地解析 MCP。关于 MCP将聚焦于几个核心问题：

- **现象观察：** 为何在 AI Agent 发展浪潮中，真正的“超级 AI 应用”尚未大规模爆发，而 **MCP 协议却率先引起广泛关注**？协议在这个阶段扮演了怎样的关键角色？
- **功能必要：** 现有的大语言模型结合 API 或 Function Call（函数调用）方案是否已经足够？**为何还需要一个独立的 MCP 协议？**
- **适用边界：** **何种类型的软件和系统适合封装为 MCP 工具？何种不适合？** 换言之，我们在哪些场景下应考虑采用 MCP，哪些场景可能并非其最佳选择？
- **本质探讨：** MCP 究竟是开启 AI 新纪元的**革命性产品**，还是一个解决特定连接问题的**实用工具**？
- **产业规律：** 在传统 IT 领域（例如华为销售通信硬件），通常是**先有协议标准**（如 2G/3G/4G/5G 标准），再围绕标准构建生态和硬件产品。然而，在计算技术（CT）产业，如 AI 芯片领域（代表厂商英伟达），往往是**生态（模型、应用）先爆发式增长，而后协议才逐步形成统一**。这就引出一个矛盾：
- 

-   **在 AI Agent 领域，为何反其道而行之，在主流应用尚未普及之前，协议（MCP）就已率先登场**（Anthropic 于 2024 年 11 月发布）？这个问题非常值得深思，也是我经常与同行探讨的。

- **深度思考：** MCP 可能是在**争夺未来 AI 应用的入口控制权**？！带着这些疑问和观点，让我们一同深入解析 MCP 的来龙去脉。

## **一、 MCP 资源概览：生态初具规模**

首先了解一下 MCP 的相关资源。如前所述，Anthropic 在 2024 年 11 月发布了 MCP 协议。我们查看其官网：

- 核心内容包括 **MCP 介绍（Introduction to the MCP）**，阐述其基本概念。
- **快速入门指南（Getting Started）** 引导开发者使用。关键点在于，MCP 设计之初是为 Anthropic 的 Claude 大模型提供外部工具调用能力，开发者可遵循指南编写 MCP 描述文件。
- MCP 由专门的组织维护，提供了**丰富多样的 SDK**，涵盖 Python、Java、Kotlin、C# 等多种语言，生态支持相当完备。
- **最具价值的资源之一是 mcp-services 代码仓库（[https://github.com/modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers)）！** 该仓库展示了目前通过 MCP 协议可集成的工具库列表。令人印象深刻的是，**仓库中已集成了超过 500 个工具**，覆盖领域非常广泛。这表明 MCP 社区活跃且资源丰富。

## **二、 MCP 的本质：模型与应用的“连接器”**

让我们回到起点，看看 **MCP 是如何出现的（基础介绍）**。

Anthropic 在 2024 年 11 月推出 MCP，初衷是服务于其 Claude 模型。想象一下示意图：一侧是 Claude 模型，另一侧是各种工具，**连接两者的纽带正是 MCP**。

**MCP 解决了什么核心问题？它定义了一套规范，用于管理“应用程序”与“大语言模型”之间的上下文（Context）交互。** 众所周知，大语言模型**擅长进行思考、推理和内容生成（即操作上下文 Context），但其本身不具备执行具体操作（Execution）的能力**——这些操作需要依赖外部应用程序来提供。为了使模型“思考”产生的上下文意图能够有效、准确地驱动外部应用程序的执行，Anthropic 设计了这个位于模型与应用之间的**协议层**。

一个恰当的类比是 **USB 接口协议**。通过 USB（例如 Type-C），计算机可以连接并驱动各种外设（键盘、鼠标、存储设备等）。同理，**MCP 的核心角色就是一个标准化、通用化的“连接协议层”。**

![](images/66e3242c71f1fbaf0f8c.png)

**MCP 的根本目的非常清晰：建立一个通用标准！** 旨在简化应用程序（APP）的开发和集成过程，实现工具调用的统一。

![](images/a8d29a620e95c04d05c6.png)

## **三、 前 MCP 时代：Function Call 的局限性**

在深入探讨 MCP 之前，有必要回顾其前身—— **Function Call（函数调用）**。这是 OpenAI 在 2023 年推出的机制，初衷是好的：使 OpenAI 的大模型能够灵活调用外部系统/工具，相当于为模型附加了一个“工具箱”（Toolkit）。当模型遇到无法直接回答的问题时，会根据预设去调用特定函数（Function），函数执行后返回结果，模型再据此生成最终回复。

![](images/d185aa211b9ebd603e0b.png)

典型应用场景：用户询问“波士顿现在天气如何？”。OpenAI 模型首先检查是否有可用的天气查询工具 API。确认后，便调用对应的 Function Call。Function Call 访问天气 API 获取数据（温度、湿度等），模型接收数据后生成自然语言回复告知用户。

**听起来合理，但其局限性在实践中日益显现：**

- **高度定制化，互操作性差：** Function Call 工具需要**针对每个大模型进行单独适配**！不同模型（A模型、B模型等）在输入参数格式、触发条件、返回值结构等方面存在差异。这意味着，应用开发者或模型提供者需要进行大量的重复性适配工作。**面对海量潜在应用，这种模型专属的对接方式效率低下，成本高昂**。（这也催生了类似火山引擎“扣子”这样的平台，旨在降低构建 Agent 的门槛。而随着 MCP 兴起，这类平台也势必需要更新底层技术）。
- **扩展性挑战：** 每增加一个新工具，通常都需要调整模型的调用接口，集成过程繁琐。
- **功能相对简单：** Function Call 在其早期阶段（2023年左右）主要支持的是查天气、定时提醒等较为单一的功能。**Function Call 特点总结：**

- 模型绑定性强：不同模型规则不同。
- 实时调用：流程直接，架构层级少。
- 易于使用，但功能范围有限。**核心痛点：协议碎片化、开发重复性高、扩展性不足！**

## **四、 MCP 的崛起：标准化“连接器”的价值**

为解决 Function Call 的上述痛点，**MCP 应运而生**！**MCP 全称 Model Context Protocol，其名称中的“Context”（上下文）点明了关键——协议交互主要依赖模型的上下文（Prompt）来传递信息。**

**MCP 的核心在于定义了一套统一的规范：** 无论是连接数据库、调用第三方 API，还是使用其他任意工具或外部资源，大语言模型（如 Claude, DeepSeek, GPT 等）都**通过同一个标准接口**完成与工具的交互。目标是实现 AI Agent 任务执行的标准化，并使开发的应用工具能够**一次适配，服务于不同的模型**（提高复用性）。

**MCP 的发展历程：**

- **2024年11月：** Anthropic 的 Claude 官方宣布支持 MCP。同期，与 Anthropic 关系密切的 AI 编程工具 Cursor（接受其投资）也迅速跟进。
- **2025年3月（引爆点）：** 关键的引爆阶段到来！国内（及全球）重要的 AI Agent 参与者（或相关概念热点）为了快速集成大量工具，采用了已在国外形成一定生态的 MCP 协议。这导致 MCP 迅速流行开来！
- **行业反应：** 甚至 OpenAI 的 CEO Sam Altman 也公开表示：鉴于 MCP 的流行度，OpenAI 未来会考虑集成 MCP 功能。**至此，MCP 在实质上已成为 AI 工具调用领域的一项重要行业标准。**

![](images/aade1f37645c847aba14.png)

**为何在 CT 领域也出现了“协议先行”？** 我认为，关键在于 **AI Agent 的核心运作模式（大模型负责认知推理，工具负责执行）已经明确。** 在这种情况下，**率先建立一个统一的连接协议层（MCP），能够有效加速整个 AI Agent 生态的成熟与应用的落地**。

**采用 MCP 的优势：**

- **大幅降低开发复杂度：** 开发者遵循 MCP 协议开发工具后，无需针对每个模型进行单独适配，显著**减少了重复性编码工作**。
- **提高工具复用率：** 按照标准 MCP 协议开发的工具，理论上能被支持 MCP 的任何模型调用，为**工具市场（Toolkit Marketplace）的诞生提供了可能**。
- **解决核心瓶颈：**
- **打破数据与应用孤岛：** 可连接本地、云端等不同环境的数据源和工具（下文 Demo 将展示文件系统工具的例子）。
- **消除开发冗余。**
- **促进工具生态的统一化发展。****直观比较：引入 MCP 后，大模型调用外部工具的过程的确变得更加简洁和轻量化。** (想象架构图：左边是多模型 A/B/C，右边是众多工具，中间由统一的 MCP 层连接，替代原先复杂混乱的专有连接线。)

**历史相似性：** 这不禁让人联想到大模型时代之前的 AI 框架竞争（PyTorch, TensorFlow, Keras, MXNet, MindSpore…）。最终出现的 **ONNX（Open Neural Network Exchange）** 标准扮演了关键角色，它是一个中间表示层（IR）协议。ONNX 实现了不同框架训练的模型在不同推理引擎上的部署转换。如今 ONNX 已成为事实标准。**而当前的 MCP，其角色与当年的 ONNX 何其相似？它们都解决了关键的工程化挑战（兼容性与复用性），但并非颠覆性的技术创新（ONNX 未改变深度学习本身，MCP 也不会改变 AI Agent 的本质）——深刻的变革来自于“大模型”范式本身。**

## **五、 ZOMI 的洞察：MCP 的价值与局限**

基于实际使用体验和初步的代码分析，我来分享对 MCP 的总结与思考，尤其关注其**现存局限性与未来发展路径**：

### **核心局限性**

- **文档依赖性强：** 这是当前面临的**首要挑战**。大语言模型（LLM）基于文本（Prompt / Tokens）进行理解与操作。模型如何知道选择哪个工具、如何调用该工具？几乎完全依赖于开发者提供的**工具描述文档（docstring）** ——包括工具名称、功能说明、参数定义等。**这些描述的准确性和清晰度，直接决定了 LLM 能否正确有效地使用该工具。** 工具本身功能强大，但如果文档描述质量不佳（特别是中文文档的编写有时不够规范严谨），将极大影响 LLM 的调用效果。**工具的理解和调用效能，高度受限于 LLM 的文本理解能力以及文档质量。**
- **上下文开销大：** 随着工具生态的繁荣（如现有仓库已超500项），问题也随之而来。为了让模型知晓“当前有哪些工具可用”，这些工具的**描述信息必须作为 Prompt 的一部分输入模型**。**工具数量越多、描述越详细，消耗的上下文 Token 资源就越大！** 这对于 LLM 的**长上下文处理能力是一项严峻考验**。对于参数规模较小或长上下文能力较弱的模型，在面对庞大的工具库时，其理解和调用能力可能会大打折扣。
- **“工具执行者”定位：** 在每次调用中，LLM 只能从当前 Prompt 提供的“工具清单”中进行有限的选择和使用。它**目前无法实现类似人类“灵机一动，创造工具”的创造性智能行为**。本质上，LLM 的能力边界依然依赖于人类开发者为其“配备”的工具集。我们预见，**随着基础大模型原生能力的持续增强（尤其是在 Agent 能力方面），MCP 这类工具调用协议的相对重要性可能会逐渐降低，其部分功能将被内化于模型本身。**

### **未来的演进方向**

- **模型针对性优化：** 为何使用 Claude 体验 MCP 更为顺畅？Anthropic（Claude 的研发公司）极有可能对 Claude 模型进行了专门的**微调（Fine-tuning）或后训练（Further Pre-training）**，使其对 MCP 工具的使用方式更加精通。这体现了**原厂在特定技术栈上的先发优势和潜在的壁垒效应**。
- **模型端后训练跟进：** MCP 的核心交互基于 Prompt。理论上，**任何具备足够上下文能力的 LLM 都能适配 MCP**，前提是向其提供完善的工具描述。然而，非原生支持 MCP 的模型，其使用体验可能难以保证。因此，可以预期，未来其他模型厂商为了与 MCP 生态更好地兼容，也会对自家模型进行相关的**后训练或微调**。
- **数据驱动的能力提升：** MCP 社区或第三方组织，可能会将大量精心编写的、标准化的**工具描述文档（docstring）** 整合成一个专门的**数据集（例如用于 File Tuning / Instruction Tuning）**。模型厂商可以利用此数据集来训练或精调模型，**旨在提升模型原生理解和使用工具的能力**。这种趋势已经开始显现。
- **能力内化是终极目标？** 随着模型厂商将 MCP 相关的“工具知识”融入训练环节，**模型的 Agent 原生能力将变得更加强大和智能**。我的观点是，**MCP 的最大贡献或许在于推动基础大模型能力的进化，而非协议自身成为永恒不变的核心。** **真正的范式变革**，依然依赖于 **AI Agent 核心算法**（如规划、记忆、复杂决策）的突破以及**基础大模型认知推理能力**的实质性跃迁。MCP 的角色更侧重于提升工具调用的**工程化效率和标准化水平**。

### **回应“三连观点”：**

- **初看是协议：** 定位准确，没有问题。
- **再看是商业模式？** 我认为它**本质上仍定位为一种技术连接工具/协议标准**。
- **三看是AI应用入口？** 不可否认，它确实提供了模型与工具交互的关键通道，具备入口价值。但**真正的应用入口竞争力和价值，根基在于模型本身的能力高低**。例如，像 DeepSeek R1 这样在基础 Agent 能力上表现突出的模型，才是打开 AI 广阔应用前景的核心钥匙。MCP 本身，正如我们反复强调的，**主要负责模型与工具之间的通信管道任务**。它**不直接提升工具性能，也不直接增强模型智能，其核心价值在于提升连接的顺畅度和标准化程度**。

## **六、 MCP 应用的未来展望**

关于 MCP 更远的未来，以下结论可以先分享给大家：

- **传统软件积极拥抱：** **操作系统、数据库**等传统基础软件提供商，以及大量的**网页应用（Web Apps）**，为了迅速提供“可被大模型调用”的能力，将**积极采纳 MCP 标准**。观察 mcp-services 仓库即可发现，PostgreSQL 以及许多库已完成接入。传统领域对其有明确需求。
- **超级应用另辟蹊径：** **微信、美团、京东**等移动互联网时代的超级应用为何尚未大规模接入 MCP？因为它们自身**已然是复杂庞大的智能体（Agent）系统**！它们更可能**对外开放自身的 Agent 能力接口（API）**，或者拥抱另一种发展方向…
- **A2A（Agent-to-Agent）协议的兴起：** 超级应用更适合采纳的是 **A2A 协议**（谷歌等公司在推动）。设想一个场景：我使用的 DeepSeek Agent 需要与美团互动完成服务（例如订餐厅）。美团一方也会有自己的 Agent 负责该业务。这时，两个 Agent 之间通过 **A2A 协议直接交互、协作完成服务交易**，更符合复杂业务逻辑的处理需求。因此，我判断：**传统基础软件更适合采用 MCP 接入大模型生态；而超级应用生态的核心交互，将逐渐走向 A2A 模式。**

- **本质探讨：** MCP 究竟是开启 AI 新纪元的**革命性产品**，还是一个解决特定连接问题的**实用工具**？
- **产业规律：** 在传统 IT 领域（例如华为销售通信硬件），通常是**先有协议标准**（如 2G/3G/4G/5G 标准），再围绕标准构建生态和硬件产品。然而，在计算技术（CT）产业，如 AI 芯片领域（代表厂商英伟达），往往是**生态（模型、应用）先爆发式增长，而后协议才逐步形成统一**。这就引出一个矛盾：
- 

-   **在 AI Agent 领域，为何反其道而行之，在主流应用尚未普及之前，协议（MCP）就已率先登场**（Anthropic 于 2024 年 11 月发布）？这个问题非常值得深思，也是我经常与同行探讨的。

## **二、 MCP 的本质：模型与应用的“连接器”**

让我们回到起点，看看 **MCP 是如何出现的（基础介绍）**。

Anthropic 在 2024 年 11 月推出 MCP，初衷是服务于其 Claude 模型。想象一下示意图：一侧是 Claude 模型，另一侧是各种工具，**连接两者的纽带正是 MCP**。

**MCP 解决了什么核心问题？它定义了一套规范，用于管理“应用程序”与“大语言模型”之间的上下文（Context）交互。** 众所周知，大语言模型**擅长进行思考、推理和内容生成（即操作上下文 Context），但其本身不具备执行具体操作（Execution）的能力**——这些操作需要依赖外部应用程序来提供。为了使模型“思考”产生的上下文意图能够有效、准确地驱动外部应用程序的执行，Anthropic 设计了这个位于模型与应用之间的**协议层**。

一个恰当的类比是 **USB 接口协议**。通过 USB（例如 Type-C），计算机可以连接并驱动各种外设（键盘、鼠标、存储设备等）。同理，**MCP 的核心角色就是一个标准化、通用化的“连接协议层”。**

![](images/66e3242c71f1fbaf0f8c.png)

**MCP 的根本目的非常清晰：建立一个通用标准！** 旨在简化应用程序（APP）的开发和集成过程，实现工具调用的统一。

![](images/a8d29a620e95c04d05c6.png)

## **三、 前 MCP 时代：Function Call 的局限性**

在深入探讨 MCP 之前，有必要回顾其前身—— **Function Call（函数调用）**。这是 OpenAI 在 2023 年推出的机制，初衷是好的：使 OpenAI 的大模型能够灵活调用外部系统/工具，相当于为模型附加了一个“工具箱”（Toolkit）。当模型遇到无法直接回答的问题时，会根据预设去调用特定函数（Function），函数执行后返回结果，模型再据此生成最终回复。

![](images/d185aa211b9ebd603e0b.png)

典型应用场景：用户询问“波士顿现在天气如何？”。OpenAI 模型首先检查是否有可用的天气查询工具 API。确认后，便调用对应的 Function Call。Function Call 访问天气 API 获取数据（温度、湿度等），模型接收数据后生成自然语言回复告知用户。

**听起来合理，但其局限性在实践中日益显现：**

- **高度定制化，互操作性差：** Function Call 工具需要**针对每个大模型进行单独适配**！不同模型（A模型、B模型等）在输入参数格式、触发条件、返回值结构等方面存在差异。这意味着，应用开发者或模型提供者需要进行大量的重复性适配工作。**面对海量潜在应用，这种模型专属的对接方式效率低下，成本高昂**。（这也催生了类似火山引擎“扣子”这样的平台，旨在降低构建 Agent 的门槛。而随着 MCP 兴起，这类平台也势必需要更新底层技术）。
- **扩展性挑战：** 每增加一个新工具，通常都需要调整模型的调用接口，集成过程繁琐。
- **功能相对简单：** Function Call 在其早期阶段（2023年左右）主要支持的是查天气、定时提醒等较为单一的功能。**Function Call 特点总结：**

- 模型绑定性强：不同模型规则不同。
- 实时调用：流程直接，架构层级少。
- 易于使用，但功能范围有限。**核心痛点：协议碎片化、开发重复性高、扩展性不足！**

## **四、 MCP 的崛起：标准化“连接器”的价值**

为解决 Function Call 的上述痛点，**MCP 应运而生**！**MCP 全称 Model Context Protocol，其名称中的“Context”（上下文）点明了关键——协议交互主要依赖模型的上下文（Prompt）来传递信息。**

**MCP 的核心在于定义了一套统一的规范：** 无论是连接数据库、调用第三方 API，还是使用其他任意工具或外部资源，大语言模型（如 Claude, DeepSeek, GPT 等）都**通过同一个标准接口**完成与工具的交互。目标是实现 AI Agent 任务执行的标准化，并使开发的应用工具能够**一次适配，服务于不同的模型**（提高复用性）。

**MCP 的发展历程：**

- **2024年11月：** Anthropic 的 Claude 官方宣布支持 MCP。同期，与 Anthropic 关系密切的 AI 编程工具 Cursor（接受其投资）也迅速跟进。
- **2025年3月（引爆点）：** 关键的引爆阶段到来！国内（及全球）重要的 AI Agent 参与者（或相关概念热点）为了快速集成大量工具，采用了已在国外形成一定生态的 MCP 协议。这导致 MCP 迅速流行开来！
- **行业反应：** 甚至 OpenAI 的 CEO Sam Altman 也公开表示：鉴于 MCP 的流行度，OpenAI 未来会考虑集成 MCP 功能。**至此，MCP 在实质上已成为 AI 工具调用领域的一项重要行业标准。**

![](images/aade1f37645c847aba14.png)

**为何在 CT 领域也出现了“协议先行”？** 我认为，关键在于 **AI Agent 的核心运作模式（大模型负责认知推理，工具负责执行）已经明确。** 在这种情况下，**率先建立一个统一的连接协议层（MCP），能够有效加速整个 AI Agent 生态的成熟与应用的落地**。

**采用 MCP 的优势：**

- **大幅降低开发复杂度：** 开发者遵循 MCP 协议开发工具后，无需针对每个模型进行单独适配，显著**减少了重复性编码工作**。
- **提高工具复用率：** 按照标准 MCP 协议开发的工具，理论上能被支持 MCP 的任何模型调用，为**工具市场（Toolkit Marketplace）的诞生提供了可能**。
- **解决核心瓶颈：**
- **打破数据与应用孤岛：** 可连接本地、云端等不同环境的数据源和工具（下文 Demo 将展示文件系统工具的例子）。
- **消除开发冗余。**
- **促进工具生态的统一化发展。****直观比较：引入 MCP 后，大模型调用外部工具的过程的确变得更加简洁和轻量化。** (想象架构图：左边是多模型 A/B/C，右边是众多工具，中间由统一的 MCP 层连接，替代原先复杂混乱的专有连接线。)

**历史相似性：** 这不禁让人联想到大模型时代之前的 AI 框架竞争（PyTorch, TensorFlow, Keras, MXNet, MindSpore…）。最终出现的 **ONNX（Open Neural Network Exchange）** 标准扮演了关键角色，它是一个中间表示层（IR）协议。ONNX 实现了不同框架训练的模型在不同推理引擎上的部署转换。如今 ONNX 已成为事实标准。**而当前的 MCP，其角色与当年的 ONNX 何其相似？它们都解决了关键的工程化挑战（兼容性与复用性），但并非颠覆性的技术创新（ONNX 未改变深度学习本身，MCP 也不会改变 AI Agent 的本质）——深刻的变革来自于“大模型”范式本身。**

## **五、 ZOMI 的洞察：MCP 的价值与局限**

基于实际使用体验和初步的代码分析，我来分享对 MCP 的总结与思考，尤其关注其**现存局限性与未来发展路径**：

### **核心局限性**

- **文档依赖性强：** 这是当前面临的**首要挑战**。大语言模型（LLM）基于文本（Prompt / Tokens）进行理解与操作。模型如何知道选择哪个工具、如何调用该工具？几乎完全依赖于开发者提供的**工具描述文档（docstring）** ——包括工具名称、功能说明、参数定义等。**这些描述的准确性和清晰度，直接决定了 LLM 能否正确有效地使用该工具。** 工具本身功能强大，但如果文档描述质量不佳（特别是中文文档的编写有时不够规范严谨），将极大影响 LLM 的调用效果。**工具的理解和调用效能，高度受限于 LLM 的文本理解能力以及文档质量。**
- **上下文开销大：** 随着工具生态的繁荣（如现有仓库已超500项），问题也随之而来。为了让模型知晓“当前有哪些工具可用”，这些工具的**描述信息必须作为 Prompt 的一部分输入模型**。**工具数量越多、描述越详细，消耗的上下文 Token 资源就越大！** 这对于 LLM 的**长上下文处理能力是一项严峻考验**。对于参数规模较小或长上下文能力较弱的模型，在面对庞大的工具库时，其理解和调用能力可能会大打折扣。
- **“工具执行者”定位：** 在每次调用中，LLM 只能从当前 Prompt 提供的“工具清单”中进行有限的选择和使用。它**目前无法实现类似人类“灵机一动，创造工具”的创造性智能行为**。本质上，LLM 的能力边界依然依赖于人类开发者为其“配备”的工具集。我们预见，**随着基础大模型原生能力的持续增强（尤其是在 Agent 能力方面），MCP 这类工具调用协议的相对重要性可能会逐渐降低，其部分功能将被内化于模型本身。**

### **未来的演进方向**

- **模型针对性优化：** 为何使用 Claude 体验 MCP 更为顺畅？Anthropic（Claude 的研发公司）极有可能对 Claude 模型进行了专门的**微调（Fine-tuning）或后训练（Further Pre-training）**，使其对 MCP 工具的使用方式更加精通。这体现了**原厂在特定技术栈上的先发优势和潜在的壁垒效应**。
- **模型端后训练跟进：** MCP 的核心交互基于 Prompt。理论上，**任何具备足够上下文能力的 LLM 都能适配 MCP**，前提是向其提供完善的工具描述。然而，非原生支持 MCP 的模型，其使用体验可能难以保证。因此，可以预期，未来其他模型厂商为了与 MCP 生态更好地兼容，也会对自家模型进行相关的**后训练或微调**。
- **数据驱动的能力提升：** MCP 社区或第三方组织，可能会将大量精心编写的、标准化的**工具描述文档（docstring）** 整合成一个专门的**数据集（例如用于 File Tuning / Instruction Tuning）**。模型厂商可以利用此数据集来训练或精调模型，**旨在提升模型原生理解和使用工具的能力**。这种趋势已经开始显现。
- **能力内化是终极目标？** 随着模型厂商将 MCP 相关的“工具知识”融入训练环节，**模型的 Agent 原生能力将变得更加强大和智能**。我的观点是，**MCP 的最大贡献或许在于推动基础大模型能力的进化，而非协议自身成为永恒不变的核心。** **真正的范式变革**，依然依赖于 **AI Agent 核心算法**（如规划、记忆、复杂决策）的突破以及**基础大模型认知推理能力**的实质性跃迁。MCP 的角色更侧重于提升工具调用的**工程化效率和标准化水平**。

### **回应“三连观点”：**

- **初看是协议：** 定位准确，没有问题。
- **再看是商业模式？** 我认为它**本质上仍定位为一种技术连接工具/协议标准**。
- **三看是AI应用入口？** 不可否认，它确实提供了模型与工具交互的关键通道，具备入口价值。但**真正的应用入口竞争力和价值，根基在于模型本身的能力高低**。例如，像 DeepSeek R1 这样在基础 Agent 能力上表现突出的模型，才是打开 AI 广阔应用前景的核心钥匙。MCP 本身，正如我们反复强调的，**主要负责模型与工具之间的通信管道任务**。它**不直接提升工具性能，也不直接增强模型智能，其核心价值在于提升连接的顺畅度和标准化程度**。

## **六、 MCP 应用的未来展望**

关于 MCP 更远的未来，以下结论可以先分享给大家：

- **传统软件积极拥抱：** **操作系统、数据库**等传统基础软件提供商，以及大量的**网页应用（Web Apps）**，为了迅速提供“可被大模型调用”的能力，将**积极采纳 MCP 标准**。观察 mcp-services 仓库即可发现，PostgreSQL 以及许多库已完成接入。传统领域对其有明确需求。
- **超级应用另辟蹊径：** **微信、美团、京东**等移动互联网时代的超级应用为何尚未大规模接入 MCP？因为它们自身**已然是复杂庞大的智能体（Agent）系统**！它们更可能**对外开放自身的 Agent 能力接口（API）**，或者拥抱另一种发展方向…
- **A2A（Agent-to-Agent）协议的兴起：** 超级应用更适合采纳的是 **A2A 协议**（谷歌等公司在推动）。设想一个场景：我使用的 DeepSeek Agent 需要与美团互动完成服务（例如订餐厅）。美团一方也会有自己的 Agent 负责该业务。这时，两个 Agent 之间通过 **A2A 协议直接交互、协作完成服务交易**，更符合复杂业务逻辑的处理需求。因此，我判断：**传统基础软件更适合采用 MCP 接入大模型生态；而超级应用生态的核心交互，将逐渐走向 A2A 模式。**

同时，网络上对 MCP 也有一些不同的解读：

- **初次接触：** MCP 是一个协议。—— 这点毋庸置疑。
- **深入了解：** MCP 似乎构建了一种**新型商业模式**？
- **深度思考：** MCP 可能是在**争夺未来 AI 应用的入口控制权**？！带着这些疑问和观点，让我们一同深入解析 MCP 的来龙去脉。

## **一、 MCP 资源概览：生态初具规模**

首先了解一下 MCP 的相关资源。如前所述，Anthropic 在 2024 年 11 月发布了 MCP 协议。我们查看其官网：

- 核心内容包括 **MCP 介绍（Introduction to the MCP）**，阐述其基本概念。
- **快速入门指南（Getting Started）** 引导开发者使用。关键点在于，MCP 设计之初是为 Anthropic 的 Claude 大模型提供外部工具调用能力，开发者可遵循指南编写 MCP 描述文件。
- MCP 由专门的组织维护，提供了**丰富多样的 SDK**，涵盖 Python、Java、Kotlin、C# 等多种语言，生态支持相当完备。
- **最具价值的资源之一是 mcp-services 代码仓库（[https://github.com/modelcontextprotocol/servers](https://github.com/modelcontextprotocol/servers)）！** 该仓库展示了目前通过 MCP 协议可集成的工具库列表。令人印象深刻的是，**仓库中已集成了超过 500 个工具**，覆盖领域非常广泛。这表明 MCP 社区活跃且资源丰富。

![](images/b817be99ae4cc8158bdd.png)

**
**

**ZOMI 的初步印象是：**MCP 的核心目标明确——**使大语言模型（AI Agent）能够便捷地利用外部工具完成特定任务**。从功能目标上看，这与之前的 Function Call 或基础 AI Agent 并无本质区别。其核心价值在于引入了一套**标准化的协议方案**，旨在让大模型调用工具的过程更加**统一和高效**。

因此，**我认为 MCP 是一个相对“稳健”的创新，重心在于优化连接，而非颠覆基础模式。**基于此，我们将按以下脉络进行更深入的探讨：

- **MCP 基础解析：** 它的起源与定义。
- **MCP 的必要性：** OpenAI 的 Function Call 存在哪些不足？
- **MCP 技术拆解：** 协议的核心机制是什么？
- **ZOMI 的洞见：** 对 MCP 的价值评估与未来展望。

## **二、 MCP 的本质：模型与应用的“连接器”**

让我们回到起点，看看 **MCP 是如何出现的（基础介绍）**。

Anthropic 在 2024 年 11 月推出 MCP，初衷是服务于其 Claude 模型。想象一下示意图：一侧是 Claude 模型，另一侧是各种工具，**连接两者的纽带正是 MCP**。

**MCP 解决了什么核心问题？它定义了一套规范，用于管理“应用程序”与“大语言模型”之间的上下文（Context）交互。** 众所周知，大语言模型**擅长进行思考、推理和内容生成（即操作上下文 Context），但其本身不具备执行具体操作（Execution）的能力**——这些操作需要依赖外部应用程序来提供。为了使模型“思考”产生的上下文意图能够有效、准确地驱动外部应用程序的执行，Anthropic 设计了这个位于模型与应用之间的**协议层**。

一个恰当的类比是 **USB 接口协议**。通过 USB（例如 Type-C），计算机可以连接并驱动各种外设（键盘、鼠标、存储设备等）。同理，**MCP 的核心角色就是一个标准化、通用化的“连接协议层”。**

![](images/66e3242c71f1fbaf0f8c.png)

**MCP 的根本目的非常清晰：建立一个通用标准！** 旨在简化应用程序（APP）的开发和集成过程，实现工具调用的统一。

![](images/a8d29a620e95c04d05c6.png)

## **三、 前 MCP 时代：Function Call 的局限性**

在深入探讨 MCP 之前，有必要回顾其前身—— **Function Call（函数调用）**。这是 OpenAI 在 2023 年推出的机制，初衷是好的：使 OpenAI 的大模型能够灵活调用外部系统/工具，相当于为模型附加了一个“工具箱”（Toolkit）。当模型遇到无法直接回答的问题时，会根据预设去调用特定函数（Function），函数执行后返回结果，模型再据此生成最终回复。

![](images/d185aa211b9ebd603e0b.png)

典型应用场景：用户询问“波士顿现在天气如何？”。OpenAI 模型首先检查是否有可用的天气查询工具 API。确认后，便调用对应的 Function Call。Function Call 访问天气 API 获取数据（温度、湿度等），模型接收数据后生成自然语言回复告知用户。

**听起来合理，但其局限性在实践中日益显现：**

- **高度定制化，互操作性差：** Function Call 工具需要**针对每个大模型进行单独适配**！不同模型（A模型、B模型等）在输入参数格式、触发条件、返回值结构等方面存在差异。这意味着，应用开发者或模型提供者需要进行大量的重复性适配工作。**面对海量潜在应用，这种模型专属的对接方式效率低下，成本高昂**。（这也催生了类似火山引擎“扣子”这样的平台，旨在降低构建 Agent 的门槛。而随着 MCP 兴起，这类平台也势必需要更新底层技术）。
- **扩展性挑战：** 每增加一个新工具，通常都需要调整模型的调用接口，集成过程繁琐。
- **功能相对简单：** Function Call 在其早期阶段（2023年左右）主要支持的是查天气、定时提醒等较为单一的功能。**Function Call 特点总结：**

- 模型绑定性强：不同模型规则不同。
- 实时调用：流程直接，架构层级少。
- 易于使用，但功能范围有限。**核心痛点：协议碎片化、开发重复性高、扩展性不足！**

## **四、 MCP 的崛起：标准化“连接器”的价值**

为解决 Function Call 的上述痛点，**MCP 应运而生**！**MCP 全称 Model Context Protocol，其名称中的“Context”（上下文）点明了关键——协议交互主要依赖模型的上下文（Prompt）来传递信息。**

**MCP 的核心在于定义了一套统一的规范：** 无论是连接数据库、调用第三方 API，还是使用其他任意工具或外部资源，大语言模型（如 Claude, DeepSeek, GPT 等）都**通过同一个标准接口**完成与工具的交互。目标是实现 AI Agent 任务执行的标准化，并使开发的应用工具能够**一次适配，服务于不同的模型**（提高复用性）。

**MCP 的发展历程：**

- **2024年11月：** Anthropic 的 Claude 官方宣布支持 MCP。同期，与 Anthropic 关系密切的 AI 编程工具 Cursor（接受其投资）也迅速跟进。
- **2025年3月（引爆点）：** 关键的引爆阶段到来！国内（及全球）重要的 AI Agent 参与者（或相关概念热点）为了快速集成大量工具，采用了已在国外形成一定生态的 MCP 协议。这导致 MCP 迅速流行开来！
- **行业反应：** 甚至 OpenAI 的 CEO Sam Altman 也公开表示：鉴于 MCP 的流行度，OpenAI 未来会考虑集成 MCP 功能。**至此，MCP 在实质上已成为 AI 工具调用领域的一项重要行业标准。**

![](images/aade1f37645c847aba14.png)

**为何在 CT 领域也出现了“协议先行”？** 我认为，关键在于 **AI Agent 的核心运作模式（大模型负责认知推理，工具负责执行）已经明确。** 在这种情况下，**率先建立一个统一的连接协议层（MCP），能够有效加速整个 AI Agent 生态的成熟与应用的落地**。

**采用 MCP 的优势：**

- **大幅降低开发复杂度：** 开发者遵循 MCP 协议开发工具后，无需针对每个模型进行单独适配，显著**减少了重复性编码工作**。
- **提高工具复用率：** 按照标准 MCP 协议开发的工具，理论上能被支持 MCP 的任何模型调用，为**工具市场（Toolkit Marketplace）的诞生提供了可能**。
- **解决核心瓶颈：**
- **打破数据与应用孤岛：** 可连接本地、云端等不同环境的数据源和工具（下文 Demo 将展示文件系统工具的例子）。
- **消除开发冗余。**
- **促进工具生态的统一化发展。****直观比较：引入 MCP 后，大模型调用外部工具的过程的确变得更加简洁和轻量化。** (想象架构图：左边是多模型 A/B/C，右边是众多工具，中间由统一的 MCP 层连接，替代原先复杂混乱的专有连接线。)

**历史相似性：** 这不禁让人联想到大模型时代之前的 AI 框架竞争（PyTorch, TensorFlow, Keras, MXNet, MindSpore…）。最终出现的 **ONNX（Open Neural Network Exchange）** 标准扮演了关键角色，它是一个中间表示层（IR）协议。ONNX 实现了不同框架训练的模型在不同推理引擎上的部署转换。如今 ONNX 已成为事实标准。**而当前的 MCP，其角色与当年的 ONNX 何其相似？它们都解决了关键的工程化挑战（兼容性与复用性），但并非颠覆性的技术创新（ONNX 未改变深度学习本身，MCP 也不会改变 AI Agent 的本质）——深刻的变革来自于“大模型”范式本身。**

![](images/24b39005ca2d6592aafc.png)

## **五、 ZOMI 的洞察：MCP 的价值与局限**

基于实际使用体验和初步的代码分析，我来分享对 MCP 的总结与思考，尤其关注其**现存局限性与未来发展路径**：

### **核心局限性**

- **文档依赖性强：** 这是当前面临的**首要挑战**。大语言模型（LLM）基于文本（Prompt / Tokens）进行理解与操作。模型如何知道选择哪个工具、如何调用该工具？几乎完全依赖于开发者提供的**工具描述文档（docstring）** ——包括工具名称、功能说明、参数定义等。**这些描述的准确性和清晰度，直接决定了 LLM 能否正确有效地使用该工具。** 工具本身功能强大，但如果文档描述质量不佳（特别是中文文档的编写有时不够规范严谨），将极大影响 LLM 的调用效果。**工具的理解和调用效能，高度受限于 LLM 的文本理解能力以及文档质量。**
- **上下文开销大：** 随着工具生态的繁荣（如现有仓库已超500项），问题也随之而来。为了让模型知晓“当前有哪些工具可用”，这些工具的**描述信息必须作为 Prompt 的一部分输入模型**。**工具数量越多、描述越详细，消耗的上下文 Token 资源就越大！** 这对于 LLM 的**长上下文处理能力是一项严峻考验**。对于参数规模较小或长上下文能力较弱的模型，在面对庞大的工具库时，其理解和调用能力可能会大打折扣。
- **“工具执行者”定位：** 在每次调用中，LLM 只能从当前 Prompt 提供的“工具清单”中进行有限的选择和使用。它**目前无法实现类似人类“灵机一动，创造工具”的创造性智能行为**。本质上，LLM 的能力边界依然依赖于人类开发者为其“配备”的工具集。我们预见，**随着基础大模型原生能力的持续增强（尤其是在 Agent 能力方面），MCP 这类工具调用协议的相对重要性可能会逐渐降低，其部分功能将被内化于模型本身。**

### **未来的演进方向**

- **模型针对性优化：** 为何使用 Claude 体验 MCP 更为顺畅？Anthropic（Claude 的研发公司）极有可能对 Claude 模型进行了专门的**微调（Fine-tuning）或后训练（Further Pre-training）**，使其对 MCP 工具的使用方式更加精通。这体现了**原厂在特定技术栈上的先发优势和潜在的壁垒效应**。
- **模型端后训练跟进：** MCP 的核心交互基于 Prompt。理论上，**任何具备足够上下文能力的 LLM 都能适配 MCP**，前提是向其提供完善的工具描述。然而，非原生支持 MCP 的模型，其使用体验可能难以保证。因此，可以预期，未来其他模型厂商为了与 MCP 生态更好地兼容，也会对自家模型进行相关的**后训练或微调**。
- **数据驱动的能力提升：** MCP 社区或第三方组织，可能会将大量精心编写的、标准化的**工具描述文档（docstring）** 整合成一个专门的**数据集（例如用于 File Tuning / Instruction Tuning）**。模型厂商可以利用此数据集来训练或精调模型，**旨在提升模型原生理解和使用工具的能力**。这种趋势已经开始显现。
- **能力内化是终极目标？** 随着模型厂商将 MCP 相关的“工具知识”融入训练环节，**模型的 Agent 原生能力将变得更加强大和智能**。我的观点是，**MCP 的最大贡献或许在于推动基础大模型能力的进化，而非协议自身成为永恒不变的核心。** **真正的范式变革**，依然依赖于 **AI Agent 核心算法**（如规划、记忆、复杂决策）的突破以及**基础大模型认知推理能力**的实质性跃迁。MCP 的角色更侧重于提升工具调用的**工程化效率和标准化水平**。

### **回应“三连观点”：**

- **初看是协议：** 定位准确，没有问题。
- **再看是商业模式？** 我认为它**本质上仍定位为一种技术连接工具/协议标准**。
- **三看是AI应用入口？** 不可否认，它确实提供了模型与工具交互的关键通道，具备入口价值。但**真正的应用入口竞争力和价值，根基在于模型本身的能力高低**。例如，像 DeepSeek R1 这样在基础 Agent 能力上表现突出的模型，才是打开 AI 广阔应用前景的核心钥匙。MCP 本身，正如我们反复强调的，**主要负责模型与工具之间的通信管道任务**。它**不直接提升工具性能，也不直接增强模型智能，其核心价值在于提升连接的顺畅度和标准化程度**。

## **六、 MCP 应用的未来展望**

关于 MCP 更远的未来，以下结论可以先分享给大家：

- **传统软件积极拥抱：** **操作系统、数据库**等传统基础软件提供商，以及大量的**网页应用（Web Apps）**，为了迅速提供“可被大模型调用”的能力，将**积极采纳 MCP 标准**。观察 mcp-services 仓库即可发现，PostgreSQL 以及许多库已完成接入。传统领域对其有明确需求。
- **超级应用另辟蹊径：** **微信、美团、京东**等移动互联网时代的超级应用为何尚未大规模接入 MCP？因为它们自身**已然是复杂庞大的智能体（Agent）系统**！它们更可能**对外开放自身的 Agent 能力接口（API）**，或者拥抱另一种发展方向…
- **A2A（Agent-to-Agent）协议的兴起：** 超级应用更适合采纳的是 **A2A 协议**（谷歌等公司在推动）。设想一个场景：我使用的 DeepSeek Agent 需要与美团互动完成服务（例如订餐厅）。美团一方也会有自己的 Agent 负责该业务。这时，两个 Agent 之间通过 **A2A 协议直接交互、协作完成服务交易**，更符合复杂业务逻辑的处理需求。因此，我判断：**传统基础软件更适合采用 MCP 接入大模型生态；而超级应用生态的核心交互，将逐渐走向 A2A 模式。**
