# Hypotheses 假设判断层

> **核心原则**: 不说"我看好的AVGO" → 说"H9: 双垄断, 64%, 证伪条件是Q3兑现不了$16B"
> 
> **信息流**: inbox → 假设判断层(分类+certainty%) → theme → company
> 
> 把模糊的感觉变成可证伪的判断。错了进 false-beliefs 当错题本。

---

## 一、假设的七大维度

每条假设必须归入以下七个维度之一，用于对 incoming 信息做第一层分类：

| 维度 | 代号 | 覆盖范围 | 典型假设题型 |
|------|------|---------|------------|
| **D1 行业趋势** | `D1-Trend` | AI基础设施 Capex 周期, 供需缺口/过剩, 产能扩张节奏, 库存周期位置 | "AI服务器2026出货yoy>40%" |
| **D2 竞争壁垒** | `D2-Moat` | 护城河深度/可持续性, 市场份额趋势, 定价权变化, 客户切换成本 | "台积电CoWoS 2027前无替代品" |
| **D3 公司基本面** | `D3-Fund` | 营收/利润轨迹, 产能爬坡/良率, 客户集中度, 管理层执行 | "中芯7nm 2Q26营收qoq>12%" |
| **D4 宏观/地缘** | `D4-Macro` | 制裁/关税/出口管制, 汇率/利率, 供应链安全, 产业政策 | "BIS不在2026扩大实体清单" |
| **D5 技术路线** | `D5-Tech` | GPU vs ASIC vs 其他架构, 互联技术迭代, 先进封装演进, 制程迁移路径 | "ASIC在推理场景占比2027>40%" |
| **D6 下游需求** | `D6-Demand` | 云厂商资本开支, 企业AI采纳率, 推理需求增长, 模型scaling law有效性 | "四大云厂2026 Capex合计>$250B" |
| **D7 产业链位置** | `D7-Chain` | 价值分配(制造vs设计vs设备vs材料), 利润率迁移, 垂直整合 | "AI价值从训练硬件向推理/应用迁移" |

### 维度之间的关系

- D1(趋势) + D2(壁垒) → 回答"赛道好不好"
- D3(基本面) + D5(技术) → 回答"公司能不能吃到"
- D4(宏观) + D6(需求) → 回答"外部环境是否有利"
- D7(产业链) → 回答"钱流向哪里"

## 二、假设标准模板

每条假设按以下格式存储。**强制字段**用 `[必须]` 标注，**可选字段**用 `[可选]`。

```markdown
# H{N}.{M} [{维度代号}] {一句话可证伪判断}

- **状态**: 活跃 / 搁置 / 已证伪
- **维度**: {代号}+{中文名}，如 D1-Trend行业趋势
- **certainty%**: {N}%
- **投资方向**: {纯置信度判断，不绑定仓位}

## Kill Metric [必须]

> 证伪条件必须可观测、有具体阈值和时间窗口。触发任一条件 → certainty% 归零，移入 false-beliefs。

| 证伪条件 | 阈值 | 时间窗口 | 当前状态 |
|----------|------|----------|----------|
| {可观测条件1} | {具体阈值} | {观测周期} | 待观测 / ✅未触发 / ❌已触发 |
| {可观测条件2} | {具体阈值} | {观测周期} | 待观测 |

## 证据时间线 [必须]

> 按日期倒序，💚=支持假设 / 🔴=削弱假设 / 🟡=方向不明确

| 日期 | 证据 | 影响 | 来源标签 |
|------|------|------|----------|
| YYYY-MM-DD | {一句话证据摘要} | 🟢 / 🔴 / 🟡 | [本地] / [数据] / [网页] / [推测] |
| YYYY-MM-DD | {证据} | 🟢 / 🔴 / 🟡 | [来源] |

## 关联关系

- **关联 theme**: {主题}
- **关联 company**: {标的}
- **创建日期**: YYYY-MM-DD
- **最后更新**: YYYY-MM-DD

## 确定性变更记录 [必须]

| 日期 | certainty% 变更 | 原因 | 来源标签 |
|------|----------------|------|----------|
| YYYY-MM-DD | X% → Y% | {变更理由 + 触发事件} | [来源] |

## Kill Thesis 月度回看 [必须]

> 最近回看日期: YYYY-MM-DD。每30天至少一次，防止确认偏误。

1. 当前证据是否仍支持假设方向？如有矛盾，最核心的是什么？
2. 是否有新出现的反方证据被忽视或未纳入时间线？
3. Kill Metric 阈值的紧迫性是否变化？是否需要收紧或放宽？
```

### 来源标签体系 [必须遵守]

所有假设中的每条证据、每次变更记录，必须标注以下标签之一：

| 标签 | 含义 | 示例 |
|------|------|------|
| `[本地]` | 来自本知识库 wiki/sources/ 已沉淀内容 | `[本地] wiki/concept/AI-Capex.md` |
| `[网页]` | 来自 websearch / URL 的公开信息 | `[网页] 台积电Q2法说会` |
| `[数据]` | 来自数据库/API 的定量数据 (tushare/FMP/彭博) | `[数据] FMP: NVDA revenue` |
| `[知识星球]` | 来自知识星球的付费内容 | `[知识星球] 吴梓豪：xxx` |
| `[推测]` | 无法验证、基于推理的判断（不单独作为锚点） | `[推测] ASIC份额推断` |
| `[待验证]` | 已提出但尚待查证的事实 | `[待验证] 需要查财报确认` |

**规则**:
- 证伪条件只能用 `[数据]` 或 `[网页]` 级别的标签，不能用 `[推测]` 或 `[待验证]`
- 确定性变更上调时，触发事件必须来自新增的独立来源（不是已有证据的重新解读）
- 每条 `[推测]` 条目必须明确标注"推测"，绝不让推测伪装成事实

### 关键约束

1. **Kill Metric 必须有阈值+时间窗口** — 每条证伪条件必须回答"到什么时候，到什么程度算证伪"
2. **每个假设至少2条证据时间线** — 不能只有一条来源支撑
3. **初次创建 certainty% ≤ 60%** — 需要经过多源验证才能上调
4. **月度回看不得跳过** — 超过35天未回看，假设状态自动标记为"⚠️过期待审"

## 三、certainty% 置信度标尺

> 不绑定仓位操作，仅表示对这一判断的置信水平。

| 区间 | 级别 | 定义 | 来源要求 |
|------|------|------|---------|
| **0–29%** | 推测 | 单源传闻/直觉判断，无硬数据支撑 | ≥1个来源即可 |
| **30–49%** | 待验证 | 有初步迹象，但证据不足或不一致 | ≥1个可靠来源 + 1个交叉信号 |
| **50–69%** | 可能成立 | 多源数据方向一致，初步验证通过 | ≥2个独立来源，数据可交叉验证 |
| **70–89%** | 高置信 | 多源独立验证，数据持续吻合 | ≥3个独立来源，至少1次成功预测验证 |
| **90%+** | 基本确认 | 压倒性证据，接近事实 | 多元验证+公开数据确认，可考虑晋升为规则 |

### certainty% 变更规则

- **上调**: 必须提供新的独立证据（不能是已有证据的重新解读）
- **下调**: 任何与假设方向相反的证据出现，立即下调 ≥5%
- **证伪**: 证伪条件被触发 → certainty% 直接归0，移入 `00-system/false-beliefs.md`
- **停滞**: 30天无新增信号 → certainty% 自动 -5%（信息衰减惩罚）

## 四、候选假设自动发现机制

### 触发条件

AI 在以下场景中自动创建候选假设（文件前缀 `_candidate_`）：

1. **每日摘要扫描时** — 同一主题出现 ≥2 条独立信息
2. **research 命令执行时** — 发现模式性趋势但未经充分验证
3. **知识星球新帖入库时** — 与现有假设无匹配但有投资含义

### 候选假设 vs 正式假设

| | 候选假设 | 正式假设 |
|------|---------|---------|
| 文件名 | `_candidate_H{N}.md` | `H{N}-{简述}.md` |
| 创建方 | AI 自动 | 用户审核后转正 |
| 可信度 | certainty% 自动加 `?` 前缀 | certainty% 为确认值 |
| 能否进入投资决策 | ❌ 仅作观察 | ✅ 可参与决策 |

### 候选→正式 转正流程

```
AI 创建 _candidate_H{N}.md 
  → 在 /today 日报中列出待审核候选
  → 用户审核：确认/修改/驳回
  → 确认：去掉 _candidate_ 前缀，确认 certainty%
  → 驳回：删除或改为 sources 条目
```

## 五、多对多关系：假设 ↔ Theme ↔ Company

### 关系规则

```
一条假设 → 关联 1 个 theme（主要）+ 可选 0~N 个次要 theme
一条假设 → 关联 0~N 个 company
一个 theme → 关联 0~N 条假设
一个 company → 出现在 0~N 条假设中
```

### 示例

```
H01 [D6-Demand] 四大云厂2026 Capex>$250B
  → theme: AI算力投资
  → company: NVDA, AMD, AVGO, TSM

H02 [D5-Tech] ASIC推理份额2027>40%
  → theme: AI算力结构变化
  → company: AVGO, MRVL, BABA

H03 [D2-Moat] 台积电先进制程无替代者
  → theme: AI算力投资, 地缘供应链
  → company: TSM
```

## 六、假设生命周期

```
[候选] → [审核转正] → [活跃跟踪] → [证伪→false-beliefs]
                                  → [验证→晋升为规则]
                                  → [搁置→不再活跃]
```

- **搁置条件**: 90天无新信号，或主题环境发生根本变化
- **晋升条件**: certainty% ≥90%，且经历 ≥3 次独立验证事件，可晋升为 Rules
- **证伪**: 直接移入 `00-system/false-beliefs.md`，保留完整的变更历史作为错题本

## 七、与其它模块的接口

| 下游模块 | 消费假设的方式 |
|---------|-------------|
| **Theme** | 同一 theme 下所有假设的 certainty% 加权平均 → theme 置信度 |
| **Company** | 涉及同一公司的所有假设 → 公司多维度评估 |
| **Walls** | 假设中的证伪条件 → 触发 Walls 预警信号 |
| **Rules** | 高置信假设(≥90%) → 候选晋升为交易规则 |
| **false-beliefs** | 被证伪的假设 → 归档为错题本 |
| **/today 日报** | 当日有变更的假设 → 置顶摘要 |
| **/weekly 周报** | 所有活跃假设的 certainty% 变动 → 汇总表 |
