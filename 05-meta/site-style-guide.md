# Site Style Guide

> Last updated: 2026-07-15
> Scope: Annette Knowledge Hub / AI Investing website, Starlight pages, dashboards, and generated knowledge-base pages.

This file records the website rules that should be followed before changing pages, navigation, dashboards, or generated content. Future edits should treat this file as the default style and information-architecture contract.

## 1. Page Titles

- Dashboard page titles should only show the current topic name.
  - Good: `R1 上游兑现`
  - Good: `R2 下游回报`
  - Good: `G2 存储`
  - Avoid: `R1 上游兑现 | 投研驾驶舱`
- Knowledge pages should not render duplicate H1 titles. If frontmatter already provides the page title, remove the repeated first Markdown H1 during generation.
- Navigation labels should be short and readable. Do not expose long source filenames in the left sidebar.

## 2. Navigation

- Left navigation can keep hierarchical expansion.
- For a section overview page, the right content area should show all nested child titles across all levels, so the reader can jump directly to the lowest-level page.
- Hypothesis navigation follows the G/S/R structure:
  - `G 需求与周期`
  - `S 产业结构与价值捕获`
  - `R 业绩兑现`
- The S layer is fixed at three main cards unless the user explicitly changes the model:
  - `S1 芯片与加速器竞争结构`
  - `S2 先进制造瓶颈、先进封装和国产替代`
  - `S3 应用层的价值捕获`

## 3. Dashboard Tabs

- The dashboard entry in the left navigation should be named `投研驾驶舱`.
- The dashboard top switcher should use short topic tabs.
  - Current examples: `G2 存储`, `R1 上游兑现`, `R2 下游回报`
- Keep the switcher vertically centered with adjacent labels and controls.
- Tab text must have enough contrast against the background in both active and inactive states.

## 4. Color And Contrast

- Do not use low-contrast text on pale backgrounds.
- Avoid very light gray text on cream, beige, yellow, or pale rose backgrounds.
- Body text, table text, card descriptions, and labels must remain readable without zooming.
- For secondary text, use a muted but still readable color. If unsure, prefer darker text.
- Important labels, metric names, and table headers should use high-contrast colors.

## 5. Tables

- Tables should fit the available content width where possible.
- Avoid unnecessary horizontal scrollbars caused by oversized columns or empty right-side space.
- If a table truly needs horizontal scrolling, it should be because the data requires it, not because of layout waste.
- Date fields should not be split into unreadable fragments. Use enough column width or a more compact date format.

## 6. Cards And Panels

- Do not put pale, low-contrast text inside cards.
- Card descriptions should be readable on the selected background.
- Use cards for grouped information, repeated items, and dashboards; avoid nesting cards inside cards.
- Keep card headings concise and scannable.

## 7. Content Density

- The public website is a reading layer, not the raw archive.
- `03-raw` should generally stay outside the main website, except through source summaries or source reference boxes.
- `02-kb` and `04-output` should be structured for reading and navigation.
- Long research filenames can remain in the repository, but website labels should use shorter titles.

## 8. Source Links And Source Cards

- When a knowledge page uses raw material, preserve provenance.
- Source summary pages must show source provenance in the rendered page, not only in frontmatter.
- Every new `02-kb/sources` page should include a visible `信源信息` section near the top.
- Required source-card fields:
  - source platform or channel, such as IMA, Feishu, ZSXQ, WeRead, Podwise, WeChat, Gmail, or manual import.
  - source type, such as raw extraction, report, podcast, note, API sync, or article.
  - raw repository path, usually under `03-raw/...`.
  - original title, author or institution if known, original publish date if known, and ingest date.
  - original URL when available.
  - confidence level and whether external fact verification was performed.
- If multiple raw files are duplicate or same-source variants, list all relevant raw paths.
- If the content is only organized from source material, clearly say: `结构化整理，不等同于外部事实核验`.
- Source summary pages should link back to the original repository source when possible.
- If `02-kb` content relates to `03-raw`, show a small source-reference area rather than publishing all raw content.

## 9. Build Hygiene

- `npm run build` in `site/` should pass before considering website work complete.
- Build warnings should be investigated. If the warning comes from project configuration and can be safely fixed, fix it.
- Generated site content should be produced through `scripts/generate_site_content.py`, not manually edited in `site/src/content/docs/kb` or `site/src/content/docs/outputs`.
- Stable custom pages such as `404.md` may be generated by the script or committed directly when appropriate.

## 10. Default Review Checklist

Before finishing website changes, check:

- No duplicate page titles.
- Sidebar labels are short and readable.
- Dashboard titles follow the short-title rule.
- Text contrast is strong enough on pale backgrounds.
- Tables fit the page or scroll only when necessary.
- Section overview pages show all nested child links.
- `npm run build` passes.
